## Deepgram Aura streaming TTS

### ✅ WebSocket endpoint
`wss://api.deepgram.com/v1/speak` with `Authorization: Token <API_KEY>` or
`Authorization: Bearer <JWT>`. (Source: Deepgram API reference,
`/reference/text-to-speech/speak-streaming`.)

### ✅ Default voice
`aura-asteria-en` (NOT `aura-2-thalia-en`). Aura-2 voices like
`aura-2-thalia-en` are valid but not the default.

### ✅ Default sample rate
`24000` (24 kHz). Streaming-compatible encodings: `linear16`, `mulaw`,
`alaw`. The other encodings (mp3, opus, flac, aac) are REST-only.

### ✅ Speed parameter
Range `0.7–1.5`, default `1.0`.

### ✅ Max input per Speak frame
2000 characters for Aura-1 + Aura-2.

### ✅ Frame protocol
Client sends JSON text frames:
- `{"type": "Speak", "text": "..."}`
- `{"type": "Flush"}`
- `{"type": "Clear"}`
- `{"type": "Close"}`
- `{"type": "KeepAlive"}` (server idle-disconnects ~10s)

Server sends:
- Binary audio frames (raw PCM if encoding=linear16)
- JSON status frames: `Metadata`, `Flushed`, `Cleared`, `Warning`, `Error`

### ⚠️ Newer Deepgram Python SDK shape
Recent docs use `deepgram.speak.v1.connect(model=..., encoding=..., sample_rate=...)`
in their `DeepgramClient` SDK. The on-the-wire WebSocket protocol is
unchanged; only the SDK convenience surface evolved. Our Rust client uses
`tokio-tungstenite` directly, so we follow the wire protocol, not the SDK.

## OpenRouter API

### ✅ Base URL
`https://openrouter.ai/api/v1`. Streaming chat completions at
`POST /api/v1/chat/completions`.

### ✅ Auth
`Authorization: Bearer <OPENROUTER_API_KEY>`.

### ✅ Optional attribution headers
- `HTTP-Referer: <site URL>` — appears on openrouter.ai leaderboards.
- `X-OpenRouter-Title: <site name>` — preferred; alias `X-Title` also accepted.
- `X-OpenRouter-Categories: <marketplace categories>` — optional.

**These are OPTIONAL.** Setting them does not affect rate limits. They only
opt your app into discoverability on openrouter.ai's app rankings. For
AudioGraph: set `HTTP-Referer: https://github.com/Codeseys-Labs/audio-graph`
and `X-Title: AudioGraph` since they're cheap and provide attribution.

### ✅ Streaming
Standard OpenAI-compatible: pass `stream: true`. Server emits SSE chunks
with the same shape as OpenAI's. `stream_options` can request usage in the
final chunk.

### ✅ Provider routing
`provider: { order: [...] }` lets users prefer specific upstream providers.
Pass through unchanged from settings.

### ✅ Model discovery
`GET /api/v1/models` returns the catalog. Cite from the docs reference.

### ✅ Recommended default model name (as of 2026-05-19)
The docs example uses `openai/gpt-5.2`. For AudioGraph's first-time UX, we
should let users pick from a model dropdown rather than hardcoding a
default. If we need a fallback, `anthropic/claude-sonnet-4.5` is a good
balance of cost/quality for chat use.

### ✅ Caching
`cache_control` parameter at the top level — for Anthropic models, OpenRouter
auto-applies cache breakpoints to the last cacheable block.

### ✅ Plugins worth knowing
- `web` (real-time web search)
- `file-parser` (PDF)
- `response-healing` (auto-JSON repair)
- `context-compression` (middle-out prompt compression)

Out of scope for AudioGraph's first integration; document only.

## cpal (Rust audio playback)

### ✅ Latest version
`0.17.3`, released 2026-02-18 (per RustAudio/cpal GitHub releases page).

### ✅ Send/Sync of `cpal::Stream`
Platform-dependent. The crate exports compile-time assertions
`assert_stream_send!` and `assert_stream_sync!` so consumers must check
themselves. Issue [#818](https://github.com/RustAudio/cpal/issues/818)
confirms `Stream` is NOT `Send` today (specifically called out for AAudio
on Android; the maintainers haven't lifted this for other backends either).

**Conclusion confirmed**: cpal::Stream cannot live in `tauri::State`
directly. Use the actor pattern from `audio-playback.md`:
- Spawn a dedicated `std::thread`
- Stream lives in that thread's stack
- Communicate via `crossbeam-channel` from tokio tasks
- `tauri::State` holds an `AudioPlayerHandle { tx, cancel }`, not the Stream itself.

### ✅ Linux backend
`alsa` is the default backend; `pipewire` is an optional feature gated by
`features = ["pipewire"]`. AudioGraph already pulls `libpipewire-0.3-dev`
in CI for rsac, so we have the system library available — but cpal's
*pipewire backend* is opt-in. For first integration, default to `alsa`
(works through PipeWire's ALSA shim) and add the explicit pipewire feature
later if latency is unacceptable.

### ⚠️ Known Windows callback bug
[Issue #970](https://github.com/RustAudio/cpal/issues/970) reports CPAL's
input callback may stop firing on Windows when combined with
`std::sync::mpsc` or `ringbuf::Producer::push`. Reproduced on cpal 0.15.x +
specific audio drivers (6.0.9679.1 dated 5/7/2024). Status as of last
search: open issue, fix unclear.

**Mitigation for our case**: this is reported for INPUT callbacks. We're
doing OUTPUT. Different code path. But we should monitor — if our output
callback stops firing, this is the first place to look. Stress-test with
the integration test suite.

### ✅ ringbuf version
`ringbuf` 0.4 confirmed as the current version. SPSC ring buffer, lock-free,
designed for the producer/realtime-callback pattern.

### ✅ rubato
Already a transitive dep through rsac. Use `FftFixedIn` for 24 kHz → 48 kHz
upsampling in the producer task, never in the realtime callback.
